01What this is

CUassistant can scan an inbox a few times a day, decide which messages create a real obligation, and creates Microsoft 365 To Do tasks for them. The shipping scan path is read + create-task only.

The default mode sends every listed message to the Codex classifier; that is the source-of-truth behavior. Hybrid mode adds deterministic prefiltering to cut token use, and compare mode measures those rules against the agent before they are trusted. The classifier returns JSON — it gets no Graph tools and no credentials. Every action that touches the world is performed by the host process against a reviewable allow-list.

Near-term goal

Move the useful Microsoft 365 actions onto an approved delegated path without asking Clemson to support a custom hosted Azure app service. The current review path favors Microsoft's first-party Graph CLI client for To Do and future mail/calendar writes, with every exposed tool gated by policy/action-policy.yaml. Direct GCassistant Graph access remains a configurable provider path, but it is not the only way this narrow workflow can run.

Long-term goal

A Clemson-sanctioned platform where faculty and staff can run modest, well-scoped agents for teaching and administrative work — drafting student replies, summarizing committee threads, outlining lectures, triaging advisor inboxes. Starting narrow is how the project earns the trust and operational track record to grow that catalog responsibly.

03How it works

A scheduled host script owns all external connections and side effects. On each run it loads registered handlers and runs them in order. The only handler today is email triage.

  1. Host selects handlers. Registered capabilities load from src/handlers/; the active handler context is set before any provider call.
  2. Triage lists mail. The handler lists new Inbox message metadata through the configured provider (Microsoft Graph, the Codex Outlook connector, or the gws CLI for Gmail).
  3. Mode chooses the path. MODE=agent sends every message to the classifier. MODE=hybrid applies deterministic rules first and only classifies the residue. MODE=compare runs the agent as a benchmark and records what the rules would have done — with no side effects.
  4. Rules resolve obvious cases when enabled. In hybrid mode, local YAML/string matching skips known noise or creates a task from a stable sender/subject template. No model call.
  5. Classifier handles judgment. The host fetches and normalizes the body (stripping quoted replies and footers) and asks the classifier whether the message creates a real obligation. Codex runs as a constrained, sandboxed subprocess; the optional OpenAI path is a single tool-less JSON call.
  6. Host applies side effects. The classifier never receives Graph tools or credentials. It returns a decision; the host validates it, writes the audit row, checks for an existing task marker, and creates the To Do task only when needed.

The scan is read + create-task only. Filing happens separately, by hand, when the user completes the task in To Do. The optional MCP server is a separate path for agent clients; today it exposes read tools plus To Do create/update, while mail/calendar writes remain stubbed or policy-blocked until consent and validators are in place.

06Security posture

The design substitutes a reviewable, host-mediated boundary for heavyweight isolation. The full IT-facing analysis lives in docs/IT_REVIEW_NOTES.md; this is the summary.

Agent confinement strong
CUassistant does not give the classifier Microsoft Graph tools or credentials. For classification, Codex runs as a short-lived subprocess with a read-only sandbox, an ephemeral throwaway working directory, --ignore-rules (no local instruction files), ignored user config, and a JSON output schema. CUassistant does not attach shell, MCP, or host-side effect tools to that classifier process. A prompt-injection attempt has no approved action channel: output must match the schema, the host validates it, and the cwd is destroyed on exit.
Permission model strong
Two layers. permissions.ts holds a per-handler allow-list of the five Graph operations triage may perform; every Graph call asserts against it. Each operation must also map to an approval: none action in policy/action-policy.yaml, the authorized-use list. There is simply no scan-path code for sending, deleting, moving, or drafting mail — the absence is reviewable, not just disabled. The optional host MCP server uses the same authorized-use list before registering tools for an agent, fails closed on missing operation mappings, and validates policy constraints before write backends run.
Consent scope watch
OAuth scopes describe what a delegated token may technically permit; policy/action-policy.yaml describes what CUassistant is allowed to expose or execute. The old GCassistant app path can still carry broad Graph scopes, but the cleaner review path is the Microsoft first-party Graph CLI client for To Do and future reversible mail/calendar writes. Either way, the host enforces operation allow-lists and policy constraints before a backend call.
Data egress watch
Email bodies — which in a faculty inbox routinely include FERPA-protected student information — are normalized and sent to Codex / OpenAI for classification. Optional MCP Outlook mail/calendar reads are also Codex-mediated through the Outlook connector rather than direct local Microsoft Graph reads. That is the central compliance consideration for a campus deployment. The roadmap answer is policy-controlled local-LLM routing for high-sensitivity categories; it is intentionally not in the current build or consent ask. Codex's network egress is also not firewalled. A local-LLM classifier backend — routing high-sensitivity categories to an on-premise or managed-endpoint model instead of OpenAI — would keep email content off third-party infrastructure entirely and is the clean path to stronger data protection without changing the host-applies-all-side-effects boundary.
Secrets watch
Microsoft refresh tokens (and optionally an OPENAI_API_KEY) live in a local .env. The file is gitignored; login helpers create it with mode 0600 when new, and transfer scripts chmod it best-effort. A gitleaks pre-commit hook and CI workflow guard against accidental commits. Protecting the endpoint itself remains an endpoint-management responsibility.
Auditability strong
Decisions are append-only rows in decisions.jsonl, with fields such as sender, truncated subject, reasoning, model used, body hash, and task id when a task is created. Task creation writes a durable task-intent row before the Graph call and a terminal row after — crash-safe and idempotent via a dedupe marker. Token usage and cost land in usage.jsonl.
Dependencies watch
Runtime dependencies are deliberately minimal: yaml and @modelcontextprotocol/sdk. npm audit --omit=dev currently reports moderate findings in the MCP SDK's Express dependency chain. The MCP server is stdio-only, not an HTTP listener, so the exposure is narrower than a network service; the finding is documented and tracked for upstream remediation.
No container by design
CUassistant runs as a host script. Codex runs as a local sandboxed subprocess with an isolated temporary working directory, ignored local config/rules, and schema-bounded output; it is not Docker/container isolation. A container would add stronger filesystem and network isolation, but for a tool-less classifier the gap is narrower. Containerizing later is straightforward; it is not on the current roadmap.

Bottom line. The agent is tightly confined and the host boundary is small and auditable. The two items a reviewer should weigh deliberately are the breadth of the delegated mail/calendar consent scope relative to what the code exposes, and the egress of student email content to a third-party model — both known, both with documented mitigations and a roadmap.

07Current status

The launchd template is present, but activation is explicit — no scans run until the job is loaded. Admin consent on the GCassistant app registration is currently revoked pending IT review. The Graph CLI To Do path works where Tasks.ReadWrite is consented; mail and calendar write tools remain stubbed pending the requested Graph CLI scopes. The Gmail path is implemented through gws when configured, and the classification, cost-tracking, and audit machinery are functional and tested. The optional host MCP server currently exposes 12 policy-approved tools; task delete, mail writes, calendar writes, and scan triggering are hidden.
