Working name: TigerMCP

Clemson Enterprise MCP Gateway Proposal

A concise concept for a Clemson-governed MCP gateway that exposes approved AI tools to authenticated users, approved AI clients, and approved agentic workflows based on profile, data classification, relationship to data, and institutional approval.

Prepared: May 2026

Source: Markdown proposal in the CUassistant repository

Executive summaries

Leadership

Create a governed MCP gateway instead of allowing disconnected AI plugins, bots, agents, or local MCP servers to spread across campus.

IT, security, stewards

Use Clemson SSO/MFA, OAuth authorization, default-deny tool approval, approved client/agent registration, relationship checks, and audit logging.

Pilot teams

Start with read-only Public and Internal Use tools. Add personalized, administrative, and Confidential-data tools only after core controls are proven.

1. Core concept

MCP gives AI applications a standard way to discover and call external tools, resources, and prompts. Clemson can use this pattern to provide approved access to campus knowledge, services, and selected systems while keeping authorization, logging, and data classification under university control.

The proposal is not one large all-powerful AI server. It is a Clemson-managed gateway that controls which tools a user, client, or agent can see and use.

2. Proposed conceptual architecture

Approved AI clients and agentsChatGPT Edu, Microsoft Copilot, Clemson assistant, departmental assistants, research agents, workflow agents, future approved AI clients
↓
TigerMCP GatewaySSO/MFA, OAuth, client/agent registration, tool registry, RBAC/ABAC, data classification, DLP, logs, rate limits, human confirmation, emergency disable
↓
Domain MCP tool servicesPublic knowledge, CCIT, Canvas, advising, library, RCD, Microsoft 365, research, enrollment, finance, curriculum
↓
Systems of record and approved data sourcesClemson web content, ServiceNow, Canvas, Microsoft 365, Banner/iROAR where approved, finance/HR where approved, research systems, departmental databases
Architecture principle: AI clients and agents should not connect directly to Banner/iROAR, Canvas, finance, HR, research systems, or other sensitive systems of record. They should request approved tools through TigerMCP, and TigerMCP should enforce identity, approved client/agent status, role, relationship to data, data classification, logging, and approval rules before a tool is exposed or executed.

Sensitive tools should be segmented by data class and business function. Internal network or VPN access can be required, but identity and authorization should still be checked per request.

3. Approved clients, agents, and assistants

“Approved AI clients” should explicitly include agentic systems, not just chat interfaces.

Category Meaning Example posture
AI client User-facing approved application that hosts the conversation or workflow. ChatGPT Edu, Microsoft Copilot, Clemson web assistant.
AI assistant Clemson-facing experience built on an approved client or model. Campus help assistant, department assistant, research support assistant.
AI agent Semi-autonomous or scheduled workflow that can plan, call tools, or act on behalf of a user. Department reporting agent, ticket triage agent, scheduled enrollment-summary agent.

Agents should be registered separately from ordinary clients because they may run scheduled tasks, chain tools, or operate with less moment-by-moment user supervision.

4. Identity and authorization model

TigerMCP should use profile-conditioned tool access. The user profile should combine identity, affiliation, role, relationship to data, data classification, approved client/agent, and operation risk.

1.
Authenticate user or service identity
2.
Verify approved client or agent
3.
Evaluate profile and entitlements
4.
Filter visible tools
5.
Re-check at tool call
6.
Execute, redact, log

The visible tool list is not the security boundary. Every tool must still enforce authorization when called.

5. Data classification posture

Public

Best starting point for search and retrieval tools.

Internal Use

Suitable for early pilots with authentication, logging, and approval.

Confidential

Requires data steward approval, least privilege, relationship checks, and stronger audit.

Restricted

Default deny unless a dedicated approved environment is created.

6. Initial tool catalog

Phase Example tools Notes
1: Low-risk read-only AI guidance, public policy, CCIT KB, service catalog, library, RCD docs Public/Internal Use only
2: Personalized read-only My tickets, my calendar, my Canvas courses, instructor course-material lookup Own-data or relationship-scoped
3: Administrative and Confidential pilots Advisor support, department reports, enrollment summaries, curriculum planning, research project search Data steward and compliance approval
4: Confirmed actions Create ticket, draft email, create calendar event, draft KB article Human confirmation required

Avoid grades, payroll, financial aid, health data, disciplinary matters, legal records, unrestricted SIS access, and autonomous write operations in the first pilots.

7. Administrative role examples

Administrative roles are a strong use case because they often need summaries, trends, and planning support rather than unrestricted record-level access.

Role Example tools Guardrails
Department chair Enrollment trends, section fill rates, waitlist/demand summaries, course rotation gaps, curriculum-change status, faculty teaching-load summaries, staffing-needs planning, department budget summaries. Department-scope and aggregate-first; no individual student records, grades, payroll detail, or HR records unless separately approved.
Associate dean Cross-department enrollment, program health, curriculum pipeline, accreditation evidence, resource planning. College-scope only; aggregate reporting by default.
Business officer Budget status summaries, procurement policy lookup, grant/spend category explanations, draft justification language. Financial summaries only; no unrestricted transaction authority or payroll-level detail.
Program or graduate coordinator Course schedule gaps, applicant/admit pipeline summaries, milestone tracking, degree requirement explanations. Program-scope only; student-level access requires explicit role and relationship approval.

Example: a department chair might ask, “Which required courses are over 90% full next fall?” without receiving access to grades, financial aid, disciplinary records, or unrestricted student records.

8. Tool registry requirements

Approval metadata

Tool name/version, business owner, technical owner, data steward, data classification, review date, rollback path.

Access metadata

Allowed roles, relationship rules, required scopes, approved clients/agents, operation type, rate limits.

Security metadata

Input schema, output filtering, logging plan, DLP controls, egress rules, manifest signature or version pin.

Change control

Reapproval for changes to descriptions, schemas, scopes, data sources, or side effects.

9. Security controls

  • Clemson SSO/MFA and registered AI clients/agents only.
  • OAuth tokens with audience validation and short lifetimes.
  • Default-deny tool registry.
  • RBAC, ABAC, and relationship-based authorization.
  • Authorization at both tool discovery and execution.
  • Stronger restrictions for scheduled or semi-autonomous agents.
  • Human confirmation for sensitive reads and meaningful writes.
  • Prompt-injection defenses for retrieved content.
  • Signed or version-pinned tool manifests.
  • Segmented execution by data class and domain.
  • Parameter validation, DLP, rate limits, and egress controls.
  • Privacy-preserving audit logs.
  • Emergency disable for tools, clients, agents, groups, or connectors.

10. Governance model

TigerMCP should be governed as institutional infrastructure, not a departmental plugin project. Stakeholders should include CCIT, Information Security, data trustees/stewards, Registrar and FERPA/privacy partners, General Counsel/compliance, research computing, research compliance, accessibility, procurement, CheckIT reviewers, and system owners.

No MCP tool goes live without a named owner, data classification, approved users, approved clients/agents, relationship rules, allowed operations, logging plan, rollback plan, and review date.

11. Pilot plan

1. Discovery
Scope, stakeholders, approval path, non-goals.
2. Gateway POC
SSO, client/agent registration, tool list/call, logs.
3. Low-risk tools
Policy, AI guidance, CCIT, service catalog, library, RCD.
4. Personalized read-only
My tickets, calendar, course context.
5. Admin or Confidential pilot
One approved scoped use case.
6. Confirmed actions
Draft or submit workflows with visible confirmation.

12. Recommended decision request

Approve a limited discovery and proof-of-concept phase to form a working group, confirm the governance path, select one approved AI client and one limited agentic workflow, build a small remote gateway, and launch only Public/Internal Use read-only tools first.

Success criterion: Clemson proves a repeatable, governed pattern for safely exposing a small number of useful AI tools. Tool count is not the measure of success.
