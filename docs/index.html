01What this is

CUassistant scans an inbox a few times a day, decides which messages create a real obligation, and creates Microsoft 365 To Do tasks for them. It never moves, archives, sends, or deletes mail.

The project is a deliberately stripped-down sibling of a larger agent. Most of it is an ordinary deterministic script; an AI classifier is called only for the cases plain rules can't decide. The classifier returns JSON — it gets no mailbox tools and no credentials. Every action that touches the world is performed by the host process, against a reviewable allow-list.

Why this side branch exists.

The parent project (CUagent) is a multi-capability agent that runs in a container with its own channels and a long-running loop — the right long-term shape, and also the shape that triggered an IT security review. CUassistant is the same idea narrowed to one capability and a much smaller surface, so the work can continue and be evaluated while consent is re-established.

Near-term goal

Restore delegated admin consent on the existing GCassistant Office 365 Microsoft 365 app registration under the same scope envelope IT already reviewed (Mail.ReadWrite, Tasks.ReadWrite, Calendars.ReadWrite, Chat.Read, offline_access). The narrow, host-applied, fully-audited shape is meant to make that re-approval straightforward to evaluate.

Long-term goal

A Clemson-sanctioned platform where faculty and staff can run modest, well-scoped agents for teaching and administrative work — drafting student replies, summarizing committee threads, outlining lectures, triaging advisor inboxes. Starting narrow is how the project earns the trust and operational track record to grow that catalog responsibly.

03How it works

A scheduled host script owns all external connections and side effects. On each run it loads registered handlers and runs them in order. The only handler today is email triage.

  1. Host selects handlers. Registered capabilities load from src/handlers/; the active handler context is set before any provider call.
  2. Triage lists mail. The handler lists new Inbox message metadata through the configured provider (Microsoft Graph, or the gws CLI for Gmail).
  3. Mode chooses the path. MODE=agent sends every message to the classifier. MODE=hybrid applies deterministic rules first and only classifies the residue. MODE=compare runs the agent as a benchmark and records what the rules would have done — with no side effects.
  4. Rules resolve obvious cases. In hybrid mode, local YAML/string matching skips known noise or creates a task from a stable sender/subject template. No model call.
  5. Classifier handles judgment. The host fetches and normalizes the body (stripping quoted replies and footers) and asks the classifier whether the message creates a real obligation. Codex runs as a constrained, sandboxed subprocess; the optional OpenAI path is a single tool-less JSON call.
  6. Host applies side effects. The classifier never receives Graph tools or credentials. It returns a decision; the host validates it, writes the audit row, checks for an existing task marker, and creates the To Do task only when needed.

The scan is read + create-task only. Filing happens separately, by hand, when the user completes the task in To Do.

06Security posture

The design substitutes a reviewable, host-mediated boundary for heavyweight isolation. The full IT-facing analysis lives in docs/IT_REVIEW_NOTES.md; this is the summary.

Agent confinement strong
The classifier never receives Microsoft Graph tools or credentials. Codex runs as a short-lived subprocess with a read-only sandbox, an ephemeral throwaway working directory, --ignore-rules (no local instruction files), and a JSON output schema. It has no shell, no file tools, no MCP surface. A prompt-injection attempt has no channel to act on anything — output must match the schema and the cwd is destroyed on exit.
Permission model strong
Two layers. permissions.ts holds a per-handler allow-list of the five Graph operations triage may perform; every Graph call asserts against it. Each operation must also map to an approval: none action in policy/action-policy.yaml. There is simply no code path for sending, deleting, moving, or drafting mail — the absence is reviewable, not just disabled.
Consent scope watch
The reviewed consent envelope (Mail.ReadWrite, Tasks.ReadWrite, Calendars.ReadWrite, Chat.Read) is broader than what triage uses — the runtime refreshes only Mail + To Do scopes. Mail.ReadWrite in particular grants more than the code exercises. Mitigation is the operation allow-list above plus reviewable absence of the call sites; the residual risk is endpoint-level.
Data egress watch
Email bodies — which in a faculty inbox routinely include FERPA-protected student information — are normalized and sent to Codex / OpenAI for classification. That is the central compliance consideration for a campus deployment. The roadmap answer is policy-controlled local-LLM routing for high-sensitivity categories; it is intentionally not in the current build or consent ask. Codex's network egress is also not firewalled.
Secrets watch
A Microsoft refresh token (and optionally an OPENAI_API_KEY) live in a local .env — gitignored, written 0600 by the login helper, and guarded by a gitleaks pre-commit hook and CI workflow. Protecting the endpoint itself remains an endpoint-management responsibility.
Auditability strong
Every decision is an append-only row in decisions.jsonl (sender, truncated subject, reasoning, model used, body hash, task id). Task creation writes a durable task-intent row before the Graph call and a terminal row after — crash-safe and idempotent via a dedupe marker. Token usage and cost land in usage.jsonl.
No container by design
CUassistant runs as a host script. A container would add stronger filesystem and network isolation, but for a tool-less, schema-bounded classifier the gap is small — the sandbox's open edges (reads outside cwd, network) have no exploitation path without tools. Containerizing later is straightforward; it is not on the current roadmap.

Bottom line. The agent is tightly confined and the host boundary is small and auditable. The two items a reviewer should weigh deliberately are the breadth of the Mail.ReadWrite consent scope relative to what the code uses, and the egress of student email content to a third-party model — both known, both with documented mitigations and a roadmap.
